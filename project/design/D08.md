# Decision 8 — Build, release, and install tooling

## Decision.

Four artifacts, standardized against the sibling repositories (`embed`,
`autotune`) so a person moving between them finds the same build targets, the
same `curl … | sh` install, and the same tag-triggered release. A root
`Makefile`, an `install.sh`, a `.goreleaser.yml`, and a
`.github/workflows/release.yml`.

### `Makefile`

Mirrors the siblings' target names and meanings, and stamps the version into
the build (Conventions → Version symbol):

```make
BINARY  := login
BIN_DIR := bin
PREFIX  ?= $(HOME)/.local

.PHONY: build fmt test check install clean

build: $(BIN_DIR)/$(BINARY)

$(BIN_DIR)/$(BINARY): go.mod $(wildcard go.sum) $(shell find cmd internal -name '*.go')
	install -d $(BIN_DIR)
	go build -ldflags "-X main.version=$(shell git describe --tags --always --dirty)" -o $(BIN_DIR)/$(BINARY) ./cmd/login

fmt:
	go fmt ./...

test:
	go test ./...

check:
	go build ./...
	go vet ./...
	go test ./...
	@test -z "$$(gofmt -l .)" || { gofmt -l .; exit 1; }

install: build
	install -d $(PREFIX)/bin
	install -m 0755 $(BIN_DIR)/$(BINARY) $(PREFIX)/bin/$(BINARY)

clean:
	rm -rf $(BIN_DIR)
	go clean
```

`build` is a real file target, not phony, so a rebuild is skipped when nothing
changed. `bin/` is build output and is not tracked.

Two deliberate departures from the mirrored siblings, both preserving the
target names and their meanings, are retained from the prior tooling:

- **`go.sum` is wrapped in `$(wildcard ...)`.** This module has no dependencies
  and therefore no `go.sum`; listing it directly makes `make build` fail with
  `No rule to make target 'go.sum'`. The wildcard makes the file participate
  when it exists and be absent without error until then.
- **`check` is added.** The mirrored `fmt` target runs `go fmt ./...`, which
  *rewrites* files rather than checking them, so nothing else asserts the green
  bar. `check` runs exactly the four commands the Conventions define as green,
  with the `gofmt` step failing when any file would be reformatted. `fmt` keeps
  its rewrite behavior. `check` is a convenience wrapper, not a redefinition:
  the Conventions remain the authority on what green means.

The one new element over the prior tooling is the `-ldflags` stamp on the
`build` recipe: a local `make build` carries `git describe --tags --always
--dirty`, so `bin/login -V` reports the real checkout state rather than
the `dev` sentinel.

### `install.sh` — the `curl … | sh` installer

A POSIX `sh` script fetching a prebuilt release archive onto the user's path,
matching the siblings in shape:

```
curl -fsSL https://raw.githubusercontent.com/ikigenba/login/main/install.sh | sh
```

- Detects OS (`linux`/`darwin`) and arch (`amd64`/`arm64`) from `uname`,
  failing with a clear message on anything else.
- Install destination is `${BINDIR:-${PREFIX:-$HOME/.local}/bin}`.
- Version is `${OAUTH_LOGIN_VERSION:-latest}`. `latest` fetches
  `https://github.com/ikigenba/login/releases/latest/download/login_<os>_<arch>.tar.gz`;
  a pinned `vX.Y.Z` fetches
  `…/releases/download/vX.Y.Z/login_<os>_<arch>.tar.gz`.
- Downloads to a `mktemp -d` scratch dir cleaned on exit, untars, and
  `install -m 0755`s the binary into the destination.
- Warns on stderr if the destination is not on `$PATH`.

### `.goreleaser.yml` — the release build

goreleaser v2. Cross-compiles the versionless archives the installer fetches
and stamps the tag into the binary:

```yaml
version: 2
project_name: login

builds:
  - id: login
    main: ./cmd/login
    binary: login
    env:
      - CGO_ENABLED=0
    goos: [linux, darwin]
    goarch: [amd64, arm64]
    ldflags:
      - -s -w -X main.version={{ .Tag }}

archives:
  - id: login
    formats: [tar.gz]
    name_template: "{{ .Binary }}_{{ .Os }}_{{ .Arch }}"

checksum:
  name_template: "checksums.txt"

release:
  github:
    owner: ikigenba
    name: login

changelog:
  use: github
  sort: asc
```

The archive `name_template` is versionless so `install.sh` can fetch a stable
`releases/latest/download/login_<os>_<arch>.tar.gz` URL with no API call.
The ldflags stamp uses `{{ .Tag }}`, matching the local `git describe` scheme
(both carry the leading `v`, e.g. `v0.1.0`).

### `.github/workflows/release.yml` — the trigger

Runs goreleaser on any pushed `v*` tag and publishes a GitHub Release:

```yaml
name: release
on:
  push:
    tags: ['v*']
permissions:
  contents: write
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-go@v5
        with:
          go-version: '1.26'
      - uses: goreleaser/goreleaser-action@v6
        with:
          version: '~> v2'
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### The release model — human-initiated, tag-triggered

Cutting a release stays a **manual, human-initiated act**: a person runs
`git tag -a vX.Y.Z -m "vX.Y.Z"` on `main` and pushes it. Nothing automates the
decision to release, the version bump, or the tag itself. The automation is
purely reactive: the pushed `v*` tag triggers the workflow, which runs
goreleaser to build the cross-compiled archives and publish the GitHub Release
that `install.sh` pulls from. Because the human owns the tag, the `Makefile`
gets **no** release/tag/bump target.

## Rejected.

- **A byte-faithful mirror of the sibling `Makefile`.** `make build` would fail
  on a fresh checkout because of the missing `go.sum`, and there would be no
  target that checks the green bar. A mirror whose main target does not run is
  not worth the fidelity — hence the `$(wildcard go.sum)` and `check`
  departures.
- **Renaming targets to something more descriptive** (`all`, `verify`,
  `lint`). Fidelity to the siblings is the point; `make build test install` in
  one repo must mean the same in the others.
- **Making `fmt` a check rather than a rewrite.** Would silently change the
  meaning of a mirrored target — someone expecting their files formatted would
  get a failure. `check` carries the gate; `fmt` keeps its behavior.
- **A `release`/`tag`/cross-compile target in the `Makefile`.** Releasing is a
  human-pushed annotated tag; the cross-compilation and publishing live in
  goreleaser driven by the tag-triggered workflow, not in `make`. Keeping the
  `Makefile` free of a release target keeps the human squarely in charge of
  when and what to release.
- **Automating the tag or version bump** (a bump script, a CI job that cuts
  tags). The release trigger is deliberately a hand-cut annotated tag, matching
  how the sibling repositories operate; automating it would invent a workflow
  they do not have.
- **A per-OS/arch install path built by hand** instead of goreleaser archives.
  goreleaser already produces the exact versionless `_<os>_<arch>.tar.gz`
  assets the installer expects, cross-compiled and checksummed; reimplementing
  that in the `Makefile` or the workflow duplicates a solved problem.

## Verification.

Structural — this Decision produces build, release, and install tooling, not
program behavior. It carries **no ids**, deliberately: its proof is running the
real targets and observing the real artifacts, which is exactly the
deterministic exit condition its phase carries. The one behavior these
artifacts feed — that a stamped binary reports its injected version through
`-V` — is D6's, covered by R-9DOF-95GF; the phase's own done-bar additionally
checks the whole chain end to end by asserting `bin/login -V` equals the
current `git describe` output. Wrapping `make`, goreleaser, or a CI workflow in
a Go test would add slow, environment-dependent tests whose substrate is weaker
than simply invoking the tools, and the binary's own behavior is already
covered by D2 through D7.
